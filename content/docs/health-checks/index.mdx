# Health Checks

Vortos ships with health endpoints for load balancers, container platforms, monitoring tools, and human debugging. The framework separates “the process is alive” from “the app can safely receive traffic” so production infrastructure can make the right decision.

## Endpoints

| Endpoint | Purpose | Runs dependency checks | Fails on optional checks |
|---|---:|---:|---:|
| `GET /health/live` | Process liveness | No | No |
| `GET /health/ready` | Traffic readiness | Critical checks only | No |
| `GET /health` | Operational summary | All checks | No |

Use `/health/live` for container liveness probes. Use `/health/ready` for Kubernetes readiness probes, deployment gates, and load balancer traffic decisions. Use `/health` for dashboards and humans.

## Liveness

```bash
curl -i http://localhost:8000/health/live
```

```json
{
  "status": "ok",
  "mode": "live",
  "timestamp": "2026-05-06T10:35:21+00:00"
}
```

`/health/live` does not touch the database, Redis, MongoDB, Kafka, or any custom check. It only tells the platform that the PHP process can answer HTTP.

## Readiness

```bash
curl -i http://localhost:8000/health/ready
```

```json
{
  "status": "ok",
  "mode": "ready",
  "timestamp": "2026-05-06T10:35:21+00:00",
  "checks": {
    "database": { "status": "ok" },
    "redis": { "status": "ok" }
  }
}
```

Readiness runs only critical checks. If a critical check is degraded, Vortos returns `503 Service Unavailable`:

```json
{
  "status": "degraded",
  "mode": "ready",
  "checks": {
    "database": { "status": "degraded" }
  }
}
```

## Full Health Summary

```bash
curl -i http://localhost:8000/health
```

```json
{
  "status": "ok",
  "mode": "health",
  "timestamp": "2026-05-06T10:35:21+00:00",
  "checks": {
    "database": { "status": "ok" },
    "redis": { "status": "ok" },
    "kafka": { "status": "degraded" }
  }
}
```

`/health` runs every registered check. Optional degraded checks are reported, but they do not make the endpoint return `503`. Critical degraded checks do.

## Detail Policy

Public responses only show per-check status. Detailed responses include latency, criticality, timeout markers, and stable error codes.

Vortos reads detail behavior from environment variables:

```bash
HEALTH_DETAILS=never
HEALTH_TOKEN=change-me
HEALTH_EXPOSE_ERRORS=false
```

| Variable | Default | Values | Meaning |
|---|---|---|---|
| `HEALTH_DETAILS` | `never` | `never`, `token`, `debug`, `always` | Controls when detailed output is returned |
| `HEALTH_TOKEN` | empty | any secret string | Required when detail access uses the token path |
| `HEALTH_EXPOSE_ERRORS` | `false` | boolean | Allows raw exception messages outside dev/debug |

Policies:

| Policy | Behavior |
|---|---|
| `debug` | Detailed output in `APP_ENV=dev` or `APP_DEBUG=true`; token also works in other environments |
| `token` | Detailed output only when `X-Health-Token` matches `HEALTH_TOKEN` |
| `never` | Detailed output is disabled |
| `always` | Detailed output is always returned |

<Callout type="warn" title="Production Default">
The default is `never` — detailed output is disabled unless you explicitly set `HEALTH_DETAILS`. For monitored production systems, use `HEALTH_DETAILS=token` and set a strong `HEALTH_TOKEN`. Never use `always` on a public network.
</Callout>

## Detailed Output

```bash
curl \
  -H "X-Health-Token: change-me" \
  http://localhost:8000/health
```

```json
{
  "status": "degraded",
  "mode": "health",
  "checks": {
    "database": {
      "status": "degraded",
      "latency_ms": 14.22,
      "critical": true,
      "error_code": "database_unreachable"
    },
    "kafka": {
      "status": "degraded",
      "latency_ms": 0,
      "critical": false,
      "error_code": "kafka_extension_missing"
    }
  }
}
```

Raw exception messages are hidden outside `APP_ENV=dev` and `APP_DEBUG=true` unless `HEALTH_EXPOSE_ERRORS=true` is set:

```json
{
  "status": "degraded",
  "latency_ms": 14.22,
  "critical": true,
  "error_code": "database_unreachable",
  "error": "SQLSTATE[08006] connection failed"
}
```

## Built-In Checks

| Check | Module | Critical | What it does |
|---|---|---:|---|
| `database` | Persistence DBAL | Yes | Executes `SELECT 1` through Doctrine DBAL |
| `redis` | Cache | Yes | Sends Redis `PING` |
| `mongodb` | Persistence Mongo | Yes | Sends MongoDB admin `ping` |
| `kafka` | Messaging (Kafka driver) | No | Fetches Kafka broker metadata when Kafka transports exist |

If a module is not active, its check is not registered.

## Criticality

Critical checks decide whether `/health/ready` and `/health` return `200` or `503`. Optional checks are useful for visibility, but they do not remove the instance from service.

Use critical checks for dependencies required to serve normal traffic. Use optional checks for background systems, analytics pipelines, or integrations where the app can still handle requests.

## Timeouts

Each health check can declare a timeout in milliseconds. If a check reports latency above that timeout, Vortos marks it degraded with:

```json
{
  "status": "degraded",
  "error_code": "health_check_timeout",
  "timed_out": true
}
```

<Callout type="info" title="Timeout Semantics">
The timeout is enforced as a health classification after the check returns. Configure the underlying client timeouts too, so database, Redis, MongoDB, Kafka, and HTTP clients cannot block the request for too long.
</Callout>

## Custom Checks

Add `#[AsHealthCheck]` to any class that implements `HealthCheckInterface`.

```php title="src/Payment/Infrastructure/Health/PaymentGatewayHealthCheck.php"
<?php

declare(strict_types=1);

namespace App\Payment\Infrastructure\Health;

use Vortos\Foundation\Health\Attribute\AsHealthCheck;
use Vortos\Foundation\Health\Contract\HealthCheckInterface;
use Vortos\Foundation\Health\HealthResult;

#[AsHealthCheck(critical: false, timeoutMs: 1500)]
final class PaymentGatewayHealthCheck implements HealthCheckInterface
{
    public function __construct(private readonly PaymentGatewayClient $client) {}

    public function name(): string
    {
        return 'payment-gateway';
    }

    public function check(): HealthResult
    {
        $start = hrtime(true);

        try {
            $this->client->ping();

            return new HealthResult($this->name(), true, $this->ms($start));
        } catch (\Throwable $e) {
            return new HealthResult(
                $this->name(),
                false,
                $this->ms($start),
                $e->getMessage(),
                'payment_gateway_unreachable',
            );
        }
    }

    private function ms(int $start): float
    {
        return round((hrtime(true) - $start) / 1_000_000, 2);
    }
}
```

The compiler pass auto-registers attributed checks from `src/`. Framework modules register their own checks through their package extension.

## CLI Health Check

Run all registered health checks from the terminal with `vortos:health`:

```bash
php bin/console vortos:health
```

```
VORTOS HEALTH
────────────────────────────────────────────────────────────

  [OK]     database              1.2ms
  [OK]     redis                 0.8ms
  [WARN]   kafka                 2.5ms  [non-critical]
           broker metadata timeout

────────────────────────────────────────────────────────────
  2 passed  1 warned  0 failed
```

Exit code `0` when all critical checks pass, `1` when any critical check fails — compatible with CI pipelines and deployment gates.

### Options

```bash
# run only critical checks
php bin/console vortos:health --critical-only

# machine-readable JSON output
php bin/console vortos:health --format=json
```

JSON output:

```json
{
    "status": "ok",
    "checks": {
        "database": {
            "status": "ok",
            "latency_ms": 1.2,
            "critical": true
        },
        "redis": {
            "status": "ok",
            "latency_ms": 0.8,
            "critical": true
        }
    }
}
```

`status` is `ok` when all critical checks pass, `degraded` when any critical check fails. Optional degraded checks are reported but do not change the status to `degraded`.

## Deployment Examples

Kubernetes:

```yaml
livenessProbe:
  httpGet:
    path: /health/live
    port: 8000
readinessProbe:
  httpGet:
    path: /health/ready
    port: 8000
```

Deployment gate:

```bash
until curl -sf http://localhost:8000/health/ready; do
  sleep 2
done
```

Monitoring with details:

```bash
curl -sf \
  -H "X-Health-Token: $HEALTH_TOKEN" \
  http://localhost:8000/health
```

## Pre-flight Diagnostics

`vortos:doctor` checks configuration correctness before you boot — environment variables, module settings, and known misconfigurations. Unlike `vortos:health` (which probes live infrastructure), `vortos:doctor` checks things that are wrong before any external service is involved.

```bash
php bin/console vortos:doctor
```

```
VORTOS DOCTOR
────────────────────────────────────────────────────────────

  [OK]     messaging.replay-secret       VORTOS_REPLAY_SECRET is configured
  [WARN]   messaging.kafka-brokers       KAFKA_BROKERS is not set
           Fix: Add KAFKA_BROKERS=localhost:9092 to your .env file.

────────────────────────────────────────────────────────────
  1 passed  1 warned  0 failed
```

Exit code `0` unless errors are present. Use `--fail-on-warning` to treat warnings as failures in CI:

```bash
php bin/console vortos:doctor --fail-on-warning
```

### Built-in Checks

| Check | Module | What it checks |
|---|---|---|
| `messaging.replay-secret` | Messaging | `VORTOS_REPLAY_SECRET` is set and ≥ 32 characters |

### Custom Checks

Implement `DoctorCheckInterface` and add `#[AsDoctor]` to register it automatically:

```php title="src/Payment/Infrastructure/Doctor/PaymentGatewayConfigCheck.php"
use Vortos\Foundation\Doctor\Attribute\AsDoctor;
use Vortos\Foundation\Doctor\Contract\DoctorCheckInterface;
use Vortos\Foundation\Doctor\DoctorResult;

#[AsDoctor]
final class PaymentGatewayConfigCheck implements DoctorCheckInterface
{
    public function name(): string
    {
        return 'payment.gateway-key';
    }

    public function run(): DoctorResult
    {
        if (getenv('PAYMENT_GATEWAY_KEY') === false) {
            return DoctorResult::error(
                $this->name(),
                'PAYMENT_GATEWAY_KEY is not set — payment processing will fail',
                'Add PAYMENT_GATEWAY_KEY=<key> to your .env file.',
            );
        }

        return DoctorResult::ok($this->name(), 'PAYMENT_GATEWAY_KEY is configured');
    }
}
```

The `run()` method must return a `DoctorResult` — never throw. Three factory methods cover all cases:

```php
DoctorResult::ok('check.name', 'Everything is fine');
DoctorResult::warning('check.name', 'Optional thing missing', 'How to fix it');
DoctorResult::error('check.name', 'Required thing broken', 'How to fix it');
```
